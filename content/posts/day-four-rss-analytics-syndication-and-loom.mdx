## The Distribution Problem

Days 1 through 3 built a blog I could write on. Day 4 is about making sure anyone can actually find it.

A personal blog with no RSS feed, no analytics, and no presence on any platform is a journal, not a publication. This session adds the plumbing that turns content into reach: a way for readers to subscribe, a way for me to see who's reading, a way to cross-post to larger platforms, and a new content format that leads with video.

## RSS Feed

The simplest feature with the longest history. RSS is 25 years old and still the best way to let readers follow a blog without handing over an email address.

The implementation is a single Next.js route handler at `/feed.xml`. It reads all published posts via the same `getAllPosts()` function that powers the homepage, then templates them into RSS 2.0 XML with proper RFC 822 dates, GUID links, and category tags.

```
/feed.xml → route handler → getAllPosts() → RSS 2.0 XML
```

Three additions total:
1. **`src/app/feed.xml/route.ts`** — the route handler
2. **RSS autodiscovery link** in `<head>` — so feed readers can find it automatically
3. **RSS link in the footer** — so humans can find it too

No dependencies. RSS XML is simple enough to template as a string. The feed works in any reader — Feedly, NetNewsWire, Reeder, or a browser's built-in RSS support.

This also unlocks something we'll need later: Substack import. More on that below.

## Analytics: Two Layers

I wanted analytics I could see without leaving my admin dashboard. That turned out to require two separate systems.

### Layer 1: Vercel Web Analytics

Vercel offers built-in web analytics on the free Hobby plan. Setup is two lines:

```tsx
import { Analytics } from "@vercel/analytics/next";

// In the root layout, after </Footer>:
<Analytics />
```

This gives you the full Vercel dashboard: visitors, page views, referrers, countries, devices. Privacy-friendly — no cookies, just a hashed identifier that resets daily.

**The catch:** Vercel Web Analytics has no REST API. There's no way to query the data programmatically. You can see it in the Vercel dashboard, but you can't pull it into your own UI. This has been a community feature request since 2023 with no resolution.

For a "check the Vercel dashboard" workflow, this is fine. For "chart it in my admin," it's a dead end.

### Layer 2: Custom View Counter with Upstash Redis

So we built our own.

The architecture is simple: a client-side component fires a POST on every page load, an API route increments a counter in Redis, and an admin endpoint returns the last 30 days of data.

**Tracking:** A `PageViewTracker` component uses `usePathname()` to detect navigation and fires a request to `/api/analytics/track`. The request is fire-and-forget — if Redis is down or unconfigured, the site keeps working. Admin pages are excluded.

**Storage:** Each page view increments two keys in Upstash Redis:
- `views:2025-04-17:total` — daily total across all pages
- `views:2025-04-17:/posts/day-four-rss-analytics` — daily count per path

A Redis pipeline batches both increments plus set-membership updates (tracking which dates and paths have data) into a single round-trip.

**Admin chart:** The summary endpoint returns 30 days of daily totals plus today's top 10 pages. The admin dashboard renders this as a CSS-only bar chart — no charting library, just calculated heights and the site's design tokens.

### Setup You'll Need

This is one of those features where the code is ready but the infrastructure needs manual configuration:

1. **Vercel Web Analytics:** Go to your Vercel project dashboard → Analytics → Enable
2. **Upstash Redis:** Go to Vercel dashboard → Storage → Create a Redis database (Upstash powers this). Copy the `KV_REST_API_URL` and `KV_REST_API_TOKEN` environment variables
3. **Add env vars:** In Vercel project settings → Environment Variables, add both Redis credentials

Until the Redis credentials are set, the analytics chart in admin shows a friendly "not configured" message with setup instructions. The tracking endpoint silently no-ops. Nothing breaks.

## Medium Syndication

### Why Medium, Not Substack

I explored both platforms for cross-posting. Here's why Medium won:

**Medium** has an official Publishing API. It accepts markdown, supports a `canonicalUrl` parameter (so SEO credit stays with your original post), allows up to 5 tags, and can publish as a draft. Authentication is a self-issued integration token — no OAuth dance, no API key application process. The API is old (the docs repo was archived in 2023) but the endpoints still work.

**Substack** has no official public API for publishing. Their "Developer API" only does LinkedIn profile lookups. The unofficial route involves extracting `connect.sid` session cookies from your browser and making undocumented HTTP calls — fragile, likely to break when Substack changes their internals, and arguably violates their terms of service.

Substack does support importing content via RSS feed. So once we have an RSS feed (done), we can do a bulk import of existing posts. But that's a one-time operation, not ongoing syndication. There's no way to automatically push new posts to Substack.

For a one-person blog that publishes a few times a week, "click a button to push to Medium as a draft" is the right level of automation. Substack gets the RSS import when it makes sense.

### How It Works

The syndication flow is a single API route and an admin button:

1. On any blog post page, the admin bar now has a **"Medium"** button
2. Click it → POST to `/api/syndicate/medium` with the post slug
3. The API reads the raw MDX from GitHub, strips the frontmatter with `gray-matter`, and sends the markdown body to Medium's API
4. Medium receives the post as a **draft** with the `canonicalUrl` pointing back to vibescoder.dev
5. On success, the `mediumUrl` is written back into the post's frontmatter and committed to GitHub

Posting as a draft is deliberate. Medium's editor handles markdown well but not perfectly — code blocks, tables, and complex formatting sometimes need a touch-up. Draft mode lets me review before publishing to Medium's audience.

### Setup You'll Need

1. **Create a Medium account** if you don't have one
2. **Generate an integration token:** Medium → Settings → Security and apps → Integration tokens
3. **Get your user ID:** `curl -H "Authorization: Bearer YOUR_TOKEN" https://api.medium.com/v1/me` — note the `id` field in the response
4. **Add env vars:** `MEDIUM_TOKEN` and `MEDIUM_USER_ID` in your Vercel project settings

## Loom Video Embed

This is the feature I'm most excited about for future sessions.

The immediate implementation is simple: add an optional `loomUrl` field to post frontmatter, and when it's present, render a responsive Loom video player as the hero element at the very top of the post — above the content, above the changelog, right after the title and metadata.

```yaml
---
title: "My Post"
loomUrl: "https://www.loom.com/share/abc123def456"
---
```

The `LoomEmbed` component converts the share URL to an embed URL, wraps it in a 16:9 responsive container with the site's border styling, and adds a small "Watch" label in the site's monospace font.

### The Bigger Vision

The real power of Loom integration isn't just embedding videos — it's using them as a content source. The blog already has a voice-to-blog pipeline: record thoughts via the browser's speech API, transcribe, generate a blog post with Claude. Loom extends this pattern to video.

The workflow I want:
1. Record a Loom video — screen share, talking head, whatever
2. Loom generates a transcript automatically
3. Paste the Loom URL in the blog's recording page
4. Claude reads the transcript and generates a blog post with the video embedded at the top
5. The video is the "hero" content — readers can watch or read, their choice

This makes the blog a dual-format publication. Some topics are better explained by showing your screen. Some readers prefer video. Having both, generated from a single recording, is the kind of workflow that makes maintaining a blog sustainable.

The Loom-as-recording-source integration is queued for a future session. Today we laid the display foundation.

## What I Learned

**Vercel Analytics is great until you want your own dashboard.** The product is polished, free, and privacy-friendly. But the lack of a query API means you can't build on top of it. For anything beyond "go look at the Vercel dashboard," you need your own tracking. The two-layer approach — Vercel for the full dashboard, custom Redis for the in-app chart — gives you both without paying for either.

**Medium's API is old but functional.** The docs repo was archived in 2023, the endpoints haven't changed, and self-issued tokens still work. For cross-posting markdown with a canonical URL, it does exactly what you need. Just don't expect updates or new features.

**Platform APIs reflect platform priorities.** Medium wants content in; they give you a publishing API. Substack wants content on their platform; they give you nothing. RSS remains the universal fallback because it doesn't depend on any platform's willingness to cooperate.

**Loom changes what "writing a blog post" means.** When your blog can start from a video recording, the barrier to publishing drops. You don't need to sit down and type — you can record your screen while debugging, narrate your thought process, and let the AI turn it into a written post with the video embedded. The blog becomes a byproduct of work you're already doing.
