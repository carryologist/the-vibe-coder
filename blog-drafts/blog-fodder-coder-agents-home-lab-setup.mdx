## Session Summary

This session documented the full journey of going from a bare Coder server running on a home lab Ubuntu AI workstation to a fully configured multi-user vibe coding environment with AI agents. Everything was done conversationally through Coder Agents.

---

## What We Did

### 1. GitHub Authentication — The Long Way Around

**Goal:** Connect Coder workspaces to GitHub so agents can clone repos, commit, push, and create PRs without manual token management.

**The journey:**
- Started by exploring three options: Personal Access Tokens, SSH keys, and Coder External Auth (OAuth)
- Chose External Auth (Option 3) — configure once on the server, every user authenticates through the browser with their own GitHub account
- Created a GitHub OAuth App with the Coder tunnel URL (`https://<your-coder-tunnel>.pit-1.try.coder.app`) as the homepage and callback URL
- Skipped "Enable Device Flow" — not needed for browser-based auth

**The server configuration struggle:**
- Discovered Coder wasn't running in Docker (just using Docker for workspaces) — it was running as a manual `coder server` process from the terminal
- Found the config file at `/etc/coder.d/coder.env` but it wasn't being loaded because:
  - The file uses `VAR=value` format without `export`
  - `source` reads the file but doesn't export to child processes
- Had to `export` the variables directly in the shell before running `coder server`

**The plot twist:**
- After all the OAuth App setup, discovered the Coder version had a **built-in default GitHub provider** (`CODER_EXTERNAL_AUTH_GITHUB_DEFAULT_PROVIDER_ENABLE`) that was already enabled by default
- Didn't even need the custom OAuth App — just navigating to `/external-auth/github` triggered the authentication flow
- Lesson: check `coder server --help` before manually configuring things

### 2. Wiring GitHub Auth into the Workspace Template

**Problem:** Even after authenticating, workspaces didn't automatically have GitHub credentials available. The external auth token existed but nothing told `git` or `gh` to use it.

**Solution — Template changes to `main.tf`:**

Added a `coder_external_auth` data source:
```hcl
data "coder_external_auth" "github" {
  id = "github"
}
```

Added `GITHUB_TOKEN` to the agent's environment variables:
```hcl
GITHUB_TOKEN = data.coder_external_auth.github.access_token
```

Added to the startup script:
- Git credential helper configuration (using `x-access-token` + `$GITHUB_TOKEN`)
- GitHub CLI (`gh`) installation

**Template push workflow learned:**
```bash
mkdir -p ~/coder-templates/docker
cd ~/coder-templates/docker
coder templates pull docker .
# edit main.tf
coder templates push docker
coder update my-first-workspace  # important: stop/start alone doesn't pick up new template versions
```

**Key gotcha:** `coder update` is required to apply a new template version to an existing workspace. Just stopping and starting reuses the old version.

### 3. System Instructions for Agents

**Problem:** Even with GitHub fully wired up, agents would ask users to authenticate or provide tokens — they didn't know the environment was pre-configured.

**Solution:** Added system instructions in Agents → Settings → Behavior → System Instructions (Admin) that apply to all users:

Key instructions included:
- GitHub access is pre-configured — never ask users to authenticate
- Use `gh` CLI for all GitHub operations
- Always commit and push — workspaces are ephemeral, GitHub is the source of truth
- Bias toward action — build first, ask questions only when genuinely ambiguous
- Do the full loop: write code → install deps → test → commit → push
- Install tools with sudo as needed without asking permission
- Don't ask "would you like me to..." for obvious next steps

### 4. Workspace Template Enhancements for Vibe Coding

Added to the startup script in `main.tf`:

| Tool | Why |
|------|-----|
| GitHub CLI (`gh`) | Repo management, PRs, issues from the terminal |
| Node.js + npm | Most web projects need it (wasn't in the base image) |
| Vercel CLI | Deploy directly from the workspace — big Vercel fans |
| uv | Fast Python package manager, way better than pip for new projects |
| zip, unzip, sqlite3 | Common utilities that were missing |

All installs are idempotent (`if ! command -v ... &> /dev/null`) so they only run on first boot and don't slow down subsequent starts.

### 5. Multi-User Setup

**Goal:** Let wife use the same Coder server with her own account and GitHub credentials.

**Setup:**
- `coder users create` on the host to create her account
- She logs in at the Coder URL, creates a workspace from the Docker template
- Visits `/external-auth/github` once to link her GitHub account
- Everything else (gh, git credentials, Vercel, system instructions) is automatic from the template

## Mistakes and Gotchas Worth Writing About

1. **`source` vs `export`** — `source /etc/coder.d/coder.env` doesn't export variables to child processes. Need explicit `export` statements.
2. **Workspace template versioning** — stopping/starting a workspace doesn't update the template version. Must run `coder update <workspace>`.
3. **Agents settings vs Deployment settings** — they're in completely different places in the UI. Agents settings are for AI behavior; deployment settings are for server config.
4. **The built-in GitHub provider** — spent time creating an OAuth App before discovering Coder has a default one built in. The `--help` output had the answer all along.
5. **Nano line truncation** — long lines in nano look cut off but the data is there. Always verify with `cat` or `grep` after editing.
6. **Agent sessions don't refresh** — after template changes, need a fresh Agents session to pick up new environment variables.
